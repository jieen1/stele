# Agent 契约系统：扩展设计文档

## 文档说明

本文档记录在现有"不可变契约 + 测试翻译"插件基础上，下一阶段要实现的扩展功能的整体设计思路。文档包含：

- 架构契约（L2）
- 复杂度契约（L3）
- 事件记录系统
- 可观测性 WebUI

已经深入讨论的部分写细节；尚未细化的部分写整体思路和方向，留待后续展开。

本文档不是实现规范，是设计共识。它的目标是让回看的人能快速理解"我们为什么做这些、它们之间什么关系、每一块要解决什么问题"。

---

## 第一部分：产品哲学和定位

### 1.1 核心产品定位

本产品的卖点是**严肃契约的强拦截器**：用户通过 DSL 写出项目的关键约束（结构、架构、复杂度），插件在编译/写入/commit 时强制执行。撞契约 = 强制 agent 进入调研模式，理解为什么撞、有没有不撞的方案，调研之后要么自己解决，要么带完整论证找用户审核。

它解决的是 agent 长期独立维护代码系统时不可避免的腐化问题——人不再逐行读代码的世界里，必须有一种新的方式守住系统的核心约束。

### 1.2 三个核心信念

**信念一：人的注意力是稀缺资源，只能花在真正的决策上。**

系统的设计目标不是"防止 agent 做错事"，而是"让 agent 撞墙时强制反思，95% 情况下自己解决，5% 情况下带高质量调研找人"。每一次人介入，都应该是经过 agent 充分调研后的真问题，不是 agent 直觉抗议。

**信念二：拦截的目的不是阻止，是触发反思。**

agent 违反约束的根本原因，95% 是它没真正理解约束、没真正理解问题，用了它知识里最常见的模板套上去。拦截的目的是打断 agent 的自动补全惯性，强迫它进入深度理解模式。撞契约 = 一个信号，告诉 agent "你现在的理解很可能是错的，停下，重新调研"。

**信念三：用户投入决定产品价值。**

本产品属于"用户投入驱动"类的工具（类似 TypeScript 类型系统、合约式编程），不属于"工具自动判断"类。用户愿意写多严格的契约，系统就帮他守多严格的边界。这创造了一种正反馈：投入越多、价值越高。所有让产品向"自动判断"滑动的功能都应警惕——它们看似在增加价值，实际上在稀释产品的核心定位。

### 1.3 产品的真实边界（重要）

明确产品能解决什么、不能解决什么，避免产品定位模糊。

**能解决得很好的问题：**
- 长期维护下的结构腐化（上帝类、无限修补、复杂度爆炸）——核心节点的边界拦截
- 架构和代码偏移（依赖方向错误、绕过抽象层）——架构契约拦截
- 显式约束的破坏（数据模型、模块边界）——已有的 L1 契约

**部分解决的问题：**
- 测试糊弄（仅对契约覆盖的测试有效，其他部分无能为力）
- demo 式实现/留 TODO 跑路（不能即时拦截，但能在后续迭代撞边界时被发现）

**不解决的问题（明确不在产品范围内）：**
- 验收报告糊弄（agent 写假报告、截假图、mock 假数据）——这是过程性偷懒，不是状态性腐化
- 语义级重复代码（写法不同但做同一件事）——需要语义理解
- 错误抽象、概念漂移、隐式耦合——这些可能在未来通过其他机制扩展，但不在当前产品范围

产品当前形态聚焦在状态性的、可拦截的问题上。过程性的、需要语义理解的问题留给未来或别的工具。

## 第二部分：契约的分层模型

### 2.1 三层契约

| 层 | 类型 | 落地方式 | 验证时机 | 失败处理 |
|---|---|---|---|---|
| L1 | 结构契约（已有） | 翻译成单元测试 | 编译/CI/写入 | 调研 |
| L2 | 架构契约（新增） | 翻译成依赖图断言测试 | 编译/CI/写入 | 调研 |
| L3 | 复杂度契约（新增） | 写入时预检查（不生成测试） | commit/编译/测试 | 调研 |

**L1 和 L2 共享同一套"DSL → 测试 → 拦截"流水线**，只是测试生成器不同。L2 是 L1 自然延伸。

**L3 是独立机制**，因为它的违反不是事件而是状态，不是布尔而是连续量，硬塞进测试框架反而会让设计变形。但**对 agent 暴露的接口是统一的**——agent 看到的都是"撞契约 → 进入调研模式"，不需要知道底层实现差异。

### 2.2 撞契约后的统一处理流程

无论哪一层契约，被违反后的流程相同：

1. **拦截动作**：阻止当前操作（写入、commit）
2. **触发调研**：agent 进入强制调研模式
3. **调研产出**：结构化的产出物（必须带证据引用）
4. **两种结局**：
   - 结局 A：agent 调研后理解到位，自己用不违反契约的方案解决——**不需要人介入**
   - 结局 B：agent 调研后确认契约真的需要演化——带完整调研找用户审核

调研产出是核心，schema 见后续章节。

## 第三部分：架构契约（L2）

### 3.1 解决什么问题

架构契约本质上是"依赖图的不变量"。它解决的是 agent 在长期维护中偷偷违反架构原则的问题：
- 业务层不该依赖基础设施层，但 agent 为了快速实现塞了一段 SQL
- 应该经过 service 才能到 repository，但 agent 直接从 controller 调了
- 模块边界本来清晰，但 agent 慢慢加了跨模块调用
- 不允许循环依赖，但 agent 没意识到自己造成了循环

### 3.2 DSL 表达内容

架构契约的 DSL 需要能声明：

- **模块定义**：哪些目录/包属于哪个模块
- **层定义**：哪些模块属于哪一层（领域层、应用层、基础设施层等）
- **依赖关系**：允许的依赖方向、禁止的依赖方向
- **实现约束**：某个 interface 只能有一个实现、必须至少 N 个实现等
- **结构性禁止**：不允许循环依赖、不允许跨层调用等

### 3.3 关键设计选择：正向定义而非反向禁止

DSL 应采用**正向定义**模式：

- 正向："业务层只能依赖领域层和工具层"——任何新依赖默认非法，必须显式声明才合法
- 反向："业务层不能依赖基础设施层"——枚举禁止项，没列出的就是允许的

正向定义更严格、更能防住 agent 偷偷加新依赖，写起来更繁琐。这符合产品哲学（用户投入越多、保护越强）。

### 3.4 与现有系统的复用关系

**能直接复用：**
- DSL → 测试 的整条流水线
- 编译时生成测试的机制
- 文件写入拦截的钩子
- 测试失败 → 触发调研 的反应链
- 多语言/多测试框架的适配器层

**必须新增：**
- DSL 语法扩展（新增"模块"、"层"、"依赖关系"概念）
- 新的测试生成器（专门生成依赖图断言）
- 每种语言的 AST 解析适配——这个工作量不小，因为依赖图比"类有没有这个字段"复杂得多，要处理 import/require/include、动态导入、re-export、namespace 等

主干复用、叶子新增。最大的工程投入在每种语言的 AST 适配器上，但这个投入有意义——这些适配器以后做 L3 的部分度量也用得上。

### 3.5 实施优先级

L2 在三层契约扩展里**应该最先做**，因为：
- 和现有系统最同构、工程改造最小
- 价值最直接（架构违反是常见问题）
- 走完 L2 之后，L3 的"调研触发"部分可以复用

## 第四部分：复杂度契约（L3）

### 4.1 解决什么问题

L3 解决的是**渐进性腐化**问题。没有一次提交是"破坏架构"的，每次都只是"稍微多一点"，但累积起来代码会变得无法维护。

具体形式：上帝类、无限修补的代码、圈复杂度爆炸、模块膨胀等。

### 4.2 为什么不能套用 L1/L2 的"翻译成测试"模式

L1/L2 的违反是事件、是布尔判定、是"系统某个属性是否成立"。
L3 的违反是状态、是连续量、是"某个度量是否在阈值内"。

如果硬把 L3 翻译成测试，会有几个问题：
- 信号语义错位：测试通过/失败的意义变了
- 团队会把它当普通断言对待——"失败了那就减点行数让它通过"，恰好走向症状性修复
- 拦截时机错位：单次写入拦截"最后一根稻草"没意义，问题在前面累积

所以 L3 走另一条路：**作为 commit/编译/测试时的预检查存在，不进 CI、不生成测试**。

### 4.3 核心机制：核心节点 + 三值边界模型

**不为所有文件设置边界，只为"核心节点"设置。**

这是 L3 最关键的设计选择。理由：
- 项目里 90% 的代码是平庸代码，全量管理会让信号噪音化
- 注意力是稀缺资源，应该集中在真正承载架构和业务价值的关键节点上
- 非核心代码的腐化用通用 lint 兜底就够，不需要进入契约系统

**每个核心节点的边界用三值模型表达：**

- **current**（当前值）：系统计算的实际状态
- **ideal**（设计意图）：用户声明的理想形态
- **max**（硬边界）：再坏不能坏到哪

三值模型的含义：
- `current = ideal`：节点处于健康状态，max 是远期保险
- `ideal < current < max`：节点偏离设计意图但未失控。每次触碰应被提示"这里有未实现的设计意图"
- `current = max`：撞墙，触发调研

**这个模型才完整表达了"对节点的期望"**：单值边界（只有 max）会让 agent 觉得"只要不撞顶就行"，自然在 max 附近游荡。三值模型让 agent 知道"max 不是目标，ideal 才是"。

### 4.4 核心节点的识别

分自动和人工两条线：

**自动识别（用 graphify 或类似工具）：**
- 入度高的节点（被很多模块依赖，是基础设施型核心）
- 出度高的节点（依赖很多其他模块，是枢纽型核心，最容易长成上帝类）
- 介数中心性高的节点（依赖图上的必经之路，咽喉型核心）
- 历史变更热点（commit 历史里被频繁修改的文件）
- 当前体积异常的节点（行数、方法数、圈复杂度偏高的）

**人工补充：**
- 核心业务链路上的关键服务/聚合根
- 跨团队约定的关键接口
- 业务规则集中地（费率引擎、风控决策、订单状态机等）

自动识别会漏掉业务上的核心（工具不懂业务），人工补充会漏掉技术上的核心（人有盲区），两条线合起来才完整。

### 4.5 节点类型分层与维度选择

不是所有核心节点用同一套维度。按节点类型分层：

**业务核心类/服务类**（如 UserService、OrderProcessor）

推荐维度（3 个）：
- 行数（捕捉整体膨胀，最稳定、跨语言）
- 公开方法数（捕捉职责膨胀，注意区分公开/私有——很多工具不区分，导致"提取私有方法"反而触发告警）
- 最大方法圈复杂度（用"最大"而不是"平均"——平均会被简单 getter 拉低）

**枢纽/协调类**（高出度节点，如 Orchestrator、Facade）

推荐维度（3 个）：
- 出度（核心病灶就是依赖太广）
- 公开方法数
- 行数（兜底）

不推荐圈复杂度——好的协调类应该是线性代码、圈复杂度低，强制圈复杂度边界反而会鼓励把协调逻辑藏到被调用方。

**基础设施/工具类**（高入度节点，如核心 Repository 接口）

推荐维度（2 个）：
- 公开 API 数（基础设施类的稳定性来自接口窄）
- 入度（不限制而是监控，突然增加可能意味着过度依赖）

这类节点不需要管行数和复杂度——内部可能合理地复杂，重要的是对外承诺面要小。

**业务规则/算法核心**（如定价引擎、风控决策树）

推荐维度（2 个）：
- 行数（兜底）
- 圈复杂度（这类代码本身就是分支密集，边界要给得宽，但有上限）

不要管方法数——业务规则类经常需要很多方法表达不同规则。

**模块级维度（1 个）：**
- 模块内文件数（职责扩散的最直接信号）

### 4.6 维度选择的原则：少而准

每多一个维度，调研触发的概率不是线性增加而是接近独立事件叠加。3 个维度每个误报率 5%，整体误报率接近 14%；6 个维度接近 26%。调研有成本（agent 时间、用户注意力），高误报率会让用户开始"批量放行"，整套系统的严肃性就崩了。

每个节点 2-3 个维度、模块级 1 个维度。一个有 50 个核心节点的项目，总共 100-150 个边界值，可控。

### 4.7 边界初始化：核心导向 + 人工对话

不是全量自动扫描后让用户对每个文件签字。那种方式会签字疲劳、没有"为什么"、价值稀释。

正确流程：

**步骤 1：自动产出候选 + 数据画像**
工具扫描后，对每个候选节点输出画像（当前指标、依赖情况、最近变更等）。这是事实。

**步骤 2：节点级对话**
对每个候选节点，工具问几个问题：
- 这个节点在架构里扮演什么角色？（业务核心 / 技术基础设施 / 边界适配器 / 协调枢纽 / 其他）
- 它的合理形态应该是什么？（薄协调层 / 厚业务逻辑 / 状态持有者 / 纯函数集 / ...）
- 它当前的状态符合设计意图吗？
- 不符合的话，目标形态应该是什么？

回答后，工具基于角色和形态推导出合理的边界数字，而不是用户拍数字。把"我应该设多少行"的难题，转化为"这是个什么东西"的容易问题。

**步骤 3：沉淀理由**
每个节点的边界，连同角色定位、设计意图、当前状态评估，**作为结构化文档保存**。这就是"原因"。

这份文档的价值远超 L3 拦截本身——它是项目的**架构意图档案**。新人 onboard 看这个，code review 引用这个，未来 agent 调研引用这个。

**步骤 4：核心预算分配逻辑（高级特性，可后做）**
节点级边界之上，定义模块级和项目级的"总和边界"——单模块内所有核心节点的复杂度之和有上限、项目内所有核心节点的复杂度之和有上限。

这强迫"扩边界"必须重新分配、不能单边增长。agent 想给节点 A 扩边界，发现项目总边界吃满了，必须先论证从哪个其他节点回收——这就触发了"整个项目的优先级再讨论"，这是真正的架构决策。

### 4.8 拦截时机

**不在写入时拦截，在 commit/编译/测试时拦截。**

理由：agent 做一个任务的中途，临时让某个文件超标完全正常（先把代码倒进来再重构提取）。写入时拦截会打断正常的重构动作（重构经常需要先增后减）。

commit 是 agent 自己宣告任务完成的时刻。这时检查复杂度才有意义——你是在问 agent："你说你做完了？那让我看看你交付的状态是不是健康的。"

这个时机选择还有隐藏价值：**它把"完成"的语义和"质量"绑定了**。agent 不能再说"功能做完了，只是代码有点乱"——在系统里，乱 = 没做完。

### 4.9 度量准确性

**判断"准确"的标准：**
- 可重复：同一份代码每次算结果一样
- 可解释：算出来的数字 agent 和用户能理解
- 可比较：跨时间比较有意义
- 不可博弈：agent 不能通过等价变换把数字压下去

**各维度的具体算法：**
- **行数**：用 SLOC（去掉空行和纯注释），不是物理行。用 scc/tokei/cloc 等成熟工具，不自己实现
- **圈复杂度**：用 McCabe 经典定义（每个 if/loop/case/and/or +1）。不用"认知复杂度"——分数更合理但跨工具差异大、不可比较
- **公开方法数 / 公开 API 数**：AST 解析，按语言"公开"语义判定（Java public、Python 不下划线开头、Go 大写开头、TS export）
- **出度**：解析 import/require/include 去重计数。关键：按"目标模块"而非"目标文件"去重
- **入度**：项目级一次性反向索引

**增量统计：**
- 指标缓存：文件没变就不重算
- 依赖图增量更新：只重新解析修改文件的 import
- 跨文件指标的脏标记

**接受的盲区：**
- 动态语言（Python、Ruby、JS）的 import 解析做不到 100% 准确
- 元编程产生的代码（注解、装饰器、宏）的复杂度算不到
- TypeScript 的类型层逻辑静态分析很难做

把盲区写进文档明确告知用户，比假装覆盖然后给错误数字好得多。

### 4.10 语言适配度

**高适配（容易做好）：** Java、C#、TypeScript（带类型）、Kotlin
强类型 + 显式可见性 + 成熟 AST 工具链 + 模块概念清晰。

**中适配（需要妥协）：** Python、Ruby
可见性靠约定（下划线）、动态 import 忽略、不处理 metaclass/monkey patch。95% 场景准确。

**低适配（谨慎承诺）：** JavaScript（无类型）、Go
JS 公开/私有几乎没有静态界限，要求至少用 JSDoc 或只支持 TS。
Go 文化和"复杂度边界"理念错位，需要单独标定经验基线。

**不适配（短期不做）：** C、C++、Rust
宏让指标完全不可信、trait 和宏让"出度"和"圈复杂度"语义变形。

**工程化路径：先深做 1 种**（推荐 TypeScript 或 Java）。做到真正准确好用，再做相邻的 1 种。第一种语言的深度比覆盖广度重要 10 倍。

## 第五部分：模块重构与架构迭代的发生机制

### 5.1 区分两个尺度

- **架构迭代**：契约本身在演化（模块边界变了、依赖方向变了、核心节点角色变了）
- **模块重构**：代码在演化，契约不变（节点的内部实现变得更好，但角色、边界、接口不变）

两者的触发机制、决策权、发生方式不同。

### 5.2 模块重构的发生时机

**时机一：撞到 max 时**
agent 撞 max 进入调研，调研结论可能是"在单元内重构后再加"。这种重构是**被动的、机会驱动的**——为了完成当前任务顺带做。

**时机二：触碰 current > ideal 的节点时**
按三值模型，节点处于"偏离设计意图但未失控"状态时，agent 每次触碰会被提示。这种重构是**渐进的、童子军规则式的**——每次触碰让代码比来时更接近设计意图。累积效果可能比时机一更大。

**时机三：用户显式发起**
用户主动指令重构。agent 执行时仍受契约约束。

**关键约束**：重构动作不能改变 max 或 ideal。如果 agent 在重构中觉得"这里 ideal 设得不对应该改"，那不是重构了，是架构迭代。这条线必须画清楚。

### 5.3 架构迭代的合法触发源

(此部分思路已成型但未深入展开，留待后续细化)

可能的合法触发源：
- 调研结论指向契约演化（事件驱动、问题驱动、自下而上）
- 模式累积的诊断（最近 N 次调研涉及同一类契约的破例申请）
- 周期性的架构回顾
- 用户主动发起

## 第六部分：事件记录系统

### 6.1 设计意图

事件记录系统的目的有两个层次：

**短期（拦截系统的副产物）**：每个事件能立刻触发对应的拦截动作或调研流程。

**长期（可观测性的数据基础）**：所有事件汇总后能回答关于项目健康的问题。

设计上**为长期留接口，但短期不投入做可视化和分析**。具体讲：现在记录事件的成本应该尽量低，但 schema 设计要为未来留好空间。

### 6.2 事件 schema 的设计原则

**原则一：记录"瞬时的、不可重建的"信息，不记录"可推导的、可重建的"信息。**

事件发生那一刻有大量上下文，此时不记事后永远拿不回来。反之，事后能算出来的不必记。

**原则二：字段要能回答一类问题，不是一个问题。**

**原则三：警惕字段被 agent 自己污染。**

字段要么是系统自动产生（绝对可信），要么是 agent 产生但有外部校验锚点（如证据引用——可校验真实性）。纯主观打分类字段不要让 agent 填。

**原则四：存"判断依据"，不存"判断结果"。**

判断标准会演化，事实数据稳定不变。"严重性"应该是查询时根据当时标准算出来的派生指标。

**原则五：schema 可演化。**

给每个事件加 schema 版本号、新字段默认可选、字段一旦定义含义不能变（要改就开新字段、废弃旧字段）。

**原则六：关联性是核心，孤立事件几乎没有价值。**

事件应该形成有向图：契约违反关联调研，调研关联演化，事件关联 commit，commit 关联节点变更。

### 6.3 当前阶段记录的事件类型

第一阶段只记录三类事件，足以构建初步的可观测性：

**类型 A：契约违反事件**

核心字段：
- 事件 ID、时间戳
- 契约 ID、契约类型（L1/L2/L3）
- 触发动作（agent 想做什么——文件写入、commit、还是别的）
- 涉及的文件和行号范围
- 涉及的核心节点（如果有）
- agent 的原始意图描述（agent 当时在执行什么任务）

结构化上下文：
- 触发前的状态快照——撞契约时涉及节点的关键指标值
- 距离边界还有多远（撞 max 是越界 1 还是越界 100，意义完全不同）
- 距离上次撞同一契约的时间间隔

链接字段：
- 关联的 commit hash（如果是 commit 时触发）
- 关联的 agent 会话 ID（如果能拿到）

**类型 B：契约演化事件**

核心字段：
- 契约 ID
- 演化类型（新增 / 修改 / 删除）
- 变更前后的契约内容
- 关联的调研事件 ID（演化必须经由调研，所以一定有关联）
- 用户审批信息（谁批的、什么时候）

衍生字段：
- 该契约从建立到这次演化的"存活时间"
- 该契约历史上被撞过多少次
- 这次演化是"收紧"还是"放松"

**契约演化是项目结构性变化的最强信号**。健康项目的契约应该是稳定的，频繁演化的契约说明对应的设计还没收敛。

**类型 C：复杂度契约的度量数据**

不是事件，是时间序列：每个核心节点在每个维度上的值随时间变化。

存储建议：
- 不要每次都记完整快照（核心节点可能几百个，commit 频繁，会爆）
- 记差分 + 定期全量快照（比如每周一次全量）
- 任何历史时点的状态都能通过最近的全量加之后的差分重建

### 6.4 暂不记录的事件类型（留作未来扩展）

以下事件类型在设计上预留位置，但当前阶段不实现：

- 调研事件的详细 schema（产出、证据引用、时长等）
- commit 摘要事件（每个 commit 的全量元数据）
- 周期性度量快照（每天/每周的项目整体快照）
- agent 上下文摘要

这些可以在拦截系统稳定后逐步加入。重点是当前的 schema 设计不要把它们的位置堵死。

## 第七部分：可观测性 WebUI

### 7.1 定位

**重要：可观测性 WebUI 不是独立的产品功能，是拦截产品的展示窗口和工作报告。**

它的价值不是"独立给用户洞察"，而是**让拦截系统的工作变得可见**。

理由：
- 拦截价值即时显形（用户立刻感受到"这个东西在保护我"）
- 可观测价值累积才显形（需要看一段时间趋势才有意义）
- 拦截准确性由用户控制（用户写多严格的契约，决定多严格的拦截）
- 可观测准确性门槛极高（错判健康度/漏报真问题，整个系统价值塌方）

所以可观测 WebUI 应该是**拦截系统的工作报告**——开发成本低、和主线没冲突、增强主线说服力。

### 7.2 当前阶段的最小 WebUI

基于已记录的三类事件（契约违反、契约演化、复杂度度量），构建一个**5 分钟可读的项目健康视图**：

**视图一：契约工作概览**
- 最近一段时间的契约违反次数（按契约分类）
- 每个契约的"被撞频率趋势"——是否在变高？
- 当前哪些契约最近被频繁触发？（这些契约和现实匹配度可能在下降）

**视图二：契约演化历史**
- 历次契约演化的时间线
- 每次演化的简要描述（什么变了、为什么变）
- 演化频率——这个项目的契约稳不稳定？

**视图三：核心节点健康**
- 所有核心节点的当前三值（current/ideal/max）
- 每个节点的关键指标趋势图
- 哪些节点处于"current > ideal"的状态（偏离设计意图）
- 哪些节点的指标在加速变化

**视图四：关注列表**
- 综合上述信号，给出"建议关注"的节点和契约
- 不主动告警，让用户自己看

### 7.3 设计原则

**5 分钟可读**：用户在任何时间点都能在 5 分钟内读完。读完要 30 分钟等于又变成"读代码"了。

**按"该介入时才显著"**：项目健康时报告平淡，恶化时相应部分突出。

**展示而非判断**：呈现事实和趋势，不主动给"项目健康分数"这种综合判断。综合判断门槛高、容易失准、损害产品严肃性。

**信号密度可调**：用户能选择看简要版还是详细版。

### 7.4 不做的事

- 不做主动告警（避免变成嘈杂的告警器）
- 不做综合健康分数（准确度难保证，容易破坏信任）
- 不做智能建议（保持"用户投入驱动"的产品定位）
- 不接入第三方信号（lint、覆盖率等）——这些在未来可能集成，但当前阶段保持简单

## 第八部分：实施路线图

### 阶段一：架构契约（L2）

- DSL 扩展支持模块、层、依赖关系
- 实现依赖图断言测试的生成器
- 一种主力语言的 AST 解析适配（推荐 TypeScript 或 Java）
- 测试失败 → 调研流程的复用打通

**验证目标**：在一个真实项目里，能拦截 agent 违反架构原则的修改。

### 阶段二：复杂度契约（L3）核心

- 核心节点识别（自动扫描 + 人工补充）
- 节点级对话的初始化流程
- 三值边界模型（current/ideal/max）
- commit 时检查 max，撞 max 触发调研
- 业务核心类的三维度（行数、公开方法数、最大圈复杂度）

**验证目标**：在一个真实项目里，能拦截上帝类的形成。

### 阶段三：事件记录

- 契约违反事件的完整记录
- 契约演化事件的完整记录
- 复杂度度量的时间序列存储（差分 + 定期全量）
- schema 版本号和演化机制

**验证目标**：能查询出任何契约的违反历史、任何节点的指标轨迹。

### 阶段四：可观测性 WebUI

- 契约工作概览视图
- 契约演化历史视图
- 核心节点健康视图
- 关注列表视图

**验证目标**：用户能在 5 分钟内了解项目当前的健康状态。

### 阶段五（未细化）：扩展

可能的扩展方向，留待后续讨论决定：
- L3 的其他节点类型（枢纽/协调类、基础设施类、业务规则类）
- 模块级和项目级的总和边界（核心预算分配逻辑）
- "current > ideal"的温和提示机制
- 更多语言的适配
- 集成第三方信号（lint、覆盖率等）作为参考

## 第九部分：关键设计决策的记录

记录一些讨论过程中的关键判断，避免未来回头时重复纠结。

**决策一：L3 不生成测试。**
理由见 4.2。这是和 L1/L2 范式上最大的不同，但必须接受。

**决策二：L3 不全量管理所有文件，只管理核心节点。**
理由：注意力是稀缺资源、信号噪音的代价、产品哲学。

**决策三：边界用三值模型，不是单值。**
理由：单值会让 agent 在 max 附近游荡，三值才能表达完整的设计期望。

**决策四：拦截时机在 commit/编译/测试，不在写入。**
理由：写入时拦截会打断正常的中途状态，commit 才是"完成"的语义边界。

**决策五：调研产出必须带证据引用，引用可校验真实性。**
理由：让"agent 糊弄调研"几乎不可能——引用要么真实存在要么不存在，可以校验。

**决策六：可观测 WebUI 是拦截产品的展示窗口，不是独立产品线。**
理由：拦截价值即时显形、可观测价值累积才显形。产品策略上应优先做拦截。

**决策七：当前阶段不解决验收糊弄、语义重复、错误抽象等问题。**
理由：明确产品边界比追求覆盖所有问题更重要。这些问题留给未来或别的工具。

**决策八：第一阶段先深做一种语言。**
理由：第一种语言的深度比覆盖广度重要 10 倍。半成品产品没有忠实用户。
